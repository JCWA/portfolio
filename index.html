마이크로서비스 아키텍처 ARCHITECTURE

초기 Express 기반 모놀리식 구조에서 서비스 성장에 따라 세 가지 문제가 드러났습니다. 알림 발송 등 무거운 배치 작업이 API 응답을 저하시키고, 작은 수정도 전체 재배포가 필요했으며, 코드 구조가 개발자마다 달라 협업 비용이 증가했습니다. 부하가 큰 기능부터 분리하고 NestJS 기반으로 점진적 전환했습니다.

SERVICE역할분리 이유
API외부 사용자 트래픽, 핵심 비즈니스 로직핵심 응답 흐름 보호
Internal API서비스 간 내부 통신, 배치 트리거내외부 트래픽 분리
ConsumerSQS 기반 비동기 작업 처리알림 발송 부하 분산
ChatWebSocket 기반 실시간 채팅독립 스케일링
CMS운영자용 관리 도구독립 배포, 장애 격리
  • 배포 독립성 확보로 배포 빈도 증가
  • 서비스별 독립 스케일링 가능
  • 한 서비스 장애가 다른 서비스로 전파되지 않는 격리 구조 확보
이벤트 트래픽 대응 PERFORMANCE

K-POP 컴백·콘서트·방송 등 이벤트 시 트래픽이 급격히 증가하는 특성에 맞춰 처리 흐름을 설계했습니다.

  • SQS 기반 비동기 파이프라인 구축으로 API와 후속 작업을 분리, 트래픽 집중 시에도 핵심 응답 흐름 보호
  • Consumer가 후속 작업을 독립적으로 처리해 부하 분산
  • AWS EKS + HPA 구성으로 트래픽 급증 시 자동 스케일아웃 대응
🔧 트러블슈팅 – 블루그린 배포 시 IP pool 고갈 TROUBLESHOOTING
이벤트 대응을 위해 블루그린 배포를 진행하던 중, 신규 Pod가 대량으로 뜨면서 VPC 서브넷의 IP가 고갈되어 Pod 생성이 실패하는 문제가 발생했습니다.
  • 원인: 블루그린 방식 특성상 기존/신규 Pod 동시 존재 구간에서 IP 사용량 2배 증가, 서브넷 IP pool 한계 초과
  • 대응: 사용하지 않는 네임스페이스 Pod 정리 및 서브넷 CIDR 범위 확장으로 IP pool 확보
  • 재발 방지: 블루그린 배포 전 IP 사용량 사전 점검 프로세스 추가, 여유 IP 모니터링 알림 설정
글로벌 서비스 GLOBAL

K-POP 팬덤 특성상 사용자가 전 세계에 분포해 초기부터 글로벌 서비스를 전제로 설계했습니다.

  • 다국어: JSON 기반 번역 파일 관리, Accept-Language 헤더 기반 동적 언어 서빙 (한/영/일, 그 외 영어 폴백)
  • 타임존: API 응답을 ISO 8601 UTC 기준으로 통일, 클라이언트에서 로컬 타임존 변환 처리
  • 글로벌 다수 국가 서비스 운영
실시간 채팅 (WebSocket) REALTIME

팬들이 이벤트·컴백 순간에 실시간으로 소통할 수 있는 채팅 기능을 별도 마이크로서비스로 분리해 설계·구현했습니다.

  • 1:1 채팅, 그룹 채팅, 불특정 다수가 참여하는 오픈 채팅방 지원
  • Redis에 소켓 키를 저장해 다중 인스턴스 환경에서도 연결 상태를 일관성 있게 관리
  • 채팅 서버를 독립 서비스로 분리해 메인 API와 장애 격리 및 독립적 스케일링 확보
🔧 트러블슈팅 – 배포 시 소켓 이전 TROUBLESHOOTING
배포 시 기존 소켓 연결을 새 서버로 자연스럽게 이전하는 방법이 필요했습니다.
  • 배포 전 모든 소켓에 연결 종료 신호 전송
  • 클라이언트 재연결 시도 시 새로 뜬 서버에 자동 연결
  • 사용자 입장에서는 잠깐의 재연결만 발생하고 채팅방 상태 유지
검색 및 추천 (OpenSearch) SEARCH

팬들이 원하는 콘텐츠를 빠르게 찾고 자연스럽게 소비 흐름이 이어지도록 검색과 추천 기능을 구축했습니다.

  • OpenSearch 기반 검색 구성, 은전한닢(mecab) 한국어 형태소 분석기 적용으로 한국어 검색 품질 향상
  • 좋아요 수, 댓글 수, 팔로우 유저 행동 데이터 기반 스코어링 방식 콘텐츠 추천 구현
  • 인덱싱/조회 성능을 고려한 운영 구조 설계
🔧 트러블슈팅 – 이모지·특수문자 검색 누락 TROUBLESHOOTING
K-POP 특성상 아티스트명에 이모지·특수문자가 포함된 경우가 많은데, 형태소 분석기만으로는 토크나이징이 되지 않아 검색 결과가 누락되는 문제가 발생했습니다.
  • N-gram 분석기를 추가 적용해 이모지·특수문자가 포함된 검색어도 처리 가능하도록 개선
  • 형태소 분석기와 N-gram을 함께 사용해 일반 텍스트와 특수문자 케이스를 모두 커버
AI/LLM 연동 (뉴스 키워드 추출) AI

아티스트별로 매일 대량의 뉴스 기사가 쏟아지는데, 팬들이 핵심 소식을 빠르게 파악할 수 있도록 키워드와 요약을 자동으로 제공하는 기능을 구축했습니다.

  • 수집된 기사를 OpenAI API에 전달해 아티스트별 핵심 키워드와 한/영/일 다국어 요약을 동시 생성하는 배치 파이프라인 설계
  • JSON Schema 기반 Structured Output으로 응답 형식을 강제해 파싱 오류 방지
  • 처리 결과를 DB에 저장해 아티스트별 키워드 랭킹 순으로 서비스에 노출
🔧 트러블슈팅 – 토큰 초과 및 중복 키워드 TROUBLESHOOTING
  • 토큰 초과: 하루치 기사를 한 번에 전달하면 컨텍스트 윈도우 초과 오류 → 더 큰 컨텍스트 윈도우 모델로 교체 해결
  • 중복 키워드: 6시간 배치 운영 시 유사 기사 반복 수집으로 키워드 중복 생성 → 배치 주기 24시간으로 조정해 해소
모니터링 및 운영 체계 DEVOPS

이벤트 순간에도 서비스를 믿고 쓸 수 있도록 배포 신뢰도와 운영 관측 체계를 구축했습니다. 초기에는 장애를 사용자 제보로 뒤늦게 인지하는 경우가 있어 선제적 감지 체계가 필요했습니다.

  • Grafana/Loki 기반으로 모든 서비스 로그를 중앙에서 통합 조회
  • Sentry로 에러 수집 및 알림, Slack 실시간 알림 연동으로 이슈 즉시 인지
  • BotKube로 Kubernetes 이벤트(Pod 재시작, OOM 등) Slack 알림 연동
  • Argo CD 기반 GitOps 배포 체계로 배포 프로세스 표준화 및 변경 이력 추적
  • 블루그린 배포 방식으로 문제 발생 시 즉시 롤백 가능한 구조 확보
  • 이슈 발생 시 재현 → 원인 파악 → 대응 → 재발 방지까지 이어지는 운영 흐름 정착
